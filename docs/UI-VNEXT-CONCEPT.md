## 0. 范式主张

**从聊天室到交办台(Dispatch Desk):对话是通道,任务才是对象。**

主画像的心智是「交办 → 看着它干 → 收结果 → 能反悔」(USER-GUIDE 全篇即此叙事)。现壳层却是「聊天 + 侧栏工具箱」:一单活的四个生命阶段被打散在四处——交办=composer、看着干=右栏 agent-runs 页签、收结果=产物页签、反悔=变更页签+对话内卡片。用户必须自己在 11 个页签里拼装出「我那单活怎么样了」。v3 审计的 C 类问题(最重的功能挤在 340px 右栏)不是宽度问题,是**范式问题**:右栏页签是开发者工具箱的形状,不是任务的形状。

vNext 用三个新原语取代「会话 + 页签森林」:

| 原语 | 是什么 | 替代现有的什么 |
|---|---|---|
| **任务单** | 一单活的完整对象:委托书 + 现场 + 成果 + 台账,五态状态机(交办中/进行中/需要你/已收工/已停工) | 会话 + 右栏 6 页签的人工拼装 |
| **「需要你」收件箱** | 全局唯一打扰入口:权限请求、池审批、Agent 提问全落这里;鎏金=「等你拿主意」的专属色 | 散落的权限弹窗 / 池审批卡 / 停滞横幅 |
| **反悔柄** | 每单常驻的回退把手:台账即时间轴,单步回退 + 整单回退永远一步可达 | 变更页签 + 对话内「本轮变更」卡 |

会话(chat)降级为任务单的**通道属性**:它仍完整存在(专家可随时看原文),但不再是壳层的组织单位。

## 1. 壳层结构与导航模型

```
┌ 案头条:品牌 · 工作圈(文件夹) · 安全档 · 引擎 · 「需要你」鎏金铃 ────────────┐
├────┬────────────────────────────────────────────────────────────────┤
│ 任 │                                                                │
│ 务 │   主台(互斥单视图):交办台(首页) / 任务单 ×N / 档案               │
│ 坞 │                                                                │
├────┴────────────────────────────────────────────────────────────────┘
＋ 两个全局抽屉:「需要你」(鎏金,右滑入) / 「工房」(专家专用,Ctrl+`:终端/MCP/调试/体检)
```

- **案头条(事实层)**:只陈列四件全局事实——在哪干活(工作圈)、多守规矩(安全档)、谁在干(引擎)、有没有事等我(鎏金铃)。永不承载导航,点击只弹对应设置弹层。现顶栏的 chips 心智直接平移。
- **任务坞(对象层)**:72px 细轨,替代 288px 会话列表。每单活=一枚**瓷章**(圆形,内嵌任务首字,外圈=进度环):青花环=进行中、鎏金脉冲=需要你、静默=已收工。悬停出全名卡,点击进任务单。坞底:档案、熟活架、设置。会话列表的搜索/分组进「档案」视图,不占常驻宽度。
- **主台(工作层)**:同一时刻只有一个视图,没有二级页签森林。交办台=首页(见 §2.2);任务单=单任务全宽舞台(见 §2.3);档案=已收工任务网格。
- 导航不变量:**任何信息 ≤2 次点击可达;打扰只从鎏金铃进;开发者面从页签降为抽屉**(不再挤占常驻布局)。

## 2. 五条核心旅程

### 2.1 首跑上手:「开台」就是首页,不是覆盖层
现实现的首跑引导卡是「聊天区里的一张卡」。vNext 的交办台**原生就是引导态**:工作圈未设时,交办箱位置显示「把文件夹拖进来」大落区;引擎未就绪时,案头条引擎位红点 + 交办箱降级为「去接引擎」一步卡。三步(圈地 → 看安全档默认「每步都问」→ 点一张熟活卡试一单)全部在交办台原位完成,没有「引导结束后界面变了样」的断层——首跑态就是常态的子集。

### 2.2 交办一个任务:composer 产出的是委托书,不是消息
交办箱(大输入区)提交后先生成**交办确认卡**:干什么(AI 复述一句)、在哪(工作圈)、多守规矩(本单安全档,可单独调)、大概多久/多贵(有依据才显)。点「开工」立单,瓷章落坞,**用户可以走开**——这是与聊天范式的根本区别:发出去的不是等回复的消息,是可离场的委托。逃生舱:输入以「?」开头或点「速问」,直接问答不立单(防止闲聊被硬套任务心智)。

### 2.3 多 Agent 委托与监督:班组,不是 DAG
任务单三区:**现场**(左,宽)+ **收活台**(右:需要你/成果/台账)+ 头部(状态 + 工序进度段 + 用量)。多 Agent 在现场以**班组图**呈现——P3 画布稿(`UI-DESIGN-P3-WORKBENCH.md`)的分层布局/贝塞尔边/状态着色全盘继承,但语义换装:节点=队员(读取员/审核员…),工头一句话常驻顶部(「现在:审核员正核对第 7/10 张表」),Agent 提议的新任务以**虚线鎏金节点**直接画进图里(批准=实线化,拒绝=消失)——审批不再是图外的一张卡,而是「看得见的一道拟增工序」。插话=**递话**:选队员、说一句、下一步生效。

### 2.4 出错与反悔:停工卡 + 台账时间轴
错误不是红色气泡,是「需要你」里的**停工卡**:出了什么事(人话)+ 三个动作(再试一次 / 换个法子〔带建议〕/ 先算了)。反悔是结构性的:收活台的台账即改动时间轴,每条有回退钮;任务单头部常驻「整单回退」;已收工的任务在档案里仍保留反悔柄(检查点数据本就持久化)。安全弹层那句「都可撤销」从文案承诺升格为**空间承诺**——反悔柄永远在同一个位置。

### 2.5 沉淀技能与记忆:收工卡闭环
任务收工时现场底部出**收工卡**:成果清单 + 两个沉淀钮——「存为熟活」(把这单的委托书参数化为任务卡,自动抠出可填空位)和「记住习惯」(现有记忆起草流程)。熟活架就摆在交办台上(§v3 的「技能入口失踪案」被结构性消灭:入口即货架)。技能库/命令/记忆面板数据层不动,只换陈列。

## 3. 分级 UI 新形态:镜头制,不是删减制

v1 分级=藏页签,v3 已批评其「纸面舒适化」。vNext 改为**镜头制**:功能永远都在,分级只决定**默认用哪个镜头看现场**:
- **叙事镜头**(简易默认):现场纪要——时间线人话句子(「14:02 合并员把 10 张表并成一张,写入 汇总.xlsx〔可回退〕」),每句可展开原始卡。
- **班组图镜头**:多 Agent 任务默认;单 Agent 任务=工序条。
- **原始镜头**(专家默认可选):现有消息流/工具卡渲染器**原样嵌入**,mono 术语全量。
专家另享「工房」抽屉与快捷键全量。分级不再有「功能墙」——只有默认镜头、术语密度、密度 token(rem 体系照用,演示密度照旧)。

## 4. 青花身份再诠释

- **瓷章**:任务坞的圆形任务印,内嵌任务首字(合/改/周),外圈进度环如拉坯轮——青花身份从「界面配色」升格为「对象的物性」。
- **鎏金专义化**:金色从点缀色收紧为唯一语义——「等你拿主意」。全局只有需要你收件箱、拟增工序节点、停工卡用金。
- **青花蓝=正在干活/可信任,釉里红/赭=Provider 引擎**(沿用既有引擎色映射,不再自相矛盾)。
- 云纹水印只上装饰层(交办台空区 opacity .06),隐喻永不承载信息——防装饰主义。

## 5. 迁移评估(诚实版)

**可平移(≈70% 渲染资产)**:青花 token 全套与双主题;popover/modal/toast/palette 原语;消息流全家渲染器(→原始镜头,零改动);产物/变更/审计/文件树面板渲染逻辑(→成果区/台账/工作圈抽屉);用量看板(→案头条弹层+任务单头);agent-runs 纯函数族与 P3 画布稿(→班组图);设置模态、技能/记忆数据层;server 数据契约(sessions/SSE/agent-runs/checkpoints/audit)全部不动。

**必须重写/新建**:
1. 壳层 grid(三栏 → 案头条+坞+主台,~300 行 CSS/HTML,纯新写);
2. **任务状态派生层**(最大新逻辑):由 session 末事件 + agent-run 状态 + 未决请求折算五态。V1 可纯前端轮询拼,做准需后端 `/api/missions` 聚合端点;
3. **「需要你」跨会话聚合**(最大后端改动):现权限桥接是回合内 blocking prompt,只在当前会话出现;全局收件箱要求 server.js 把未决 prompt 落成可查询状态且不破坏「超时自动拒绝」语义——中等偏大,且 server.js 是 11k+ 行单文件,手术需串行谨慎;
4. 交办确认卡(从任务卡填空逻辑扩展)、叙事镜头折算函数(规则映射,前端纯函数)、收工卡。

**路径建议:壳层渐进置换,明确反对激进重写。** 理由:渲染器承载了全部 XSS 纪律、SSE 恢复、增量 diff、无障碍的多年边角打磨,重写=重踩全部坑;零依赖约束下没有「换框架」红利可摊销重写成本;数据契约稳定,换的只该是容器。三步走:
- **V1 坞与案头**:新壳层落地,任务单=头部+整体嵌入的现聊天视图;坞状态纯前端派生;保留「经典布局」开关。
- **V2 任务单成形**:右栏页签内容重组为现场/收活台;P3 班组图在新舞台落地;交办确认卡。
- **V3 需要你 + 叙事**:server 未决事项状态表 → 全局收件箱;叙事镜头;收工卡沉淀闭环;经典布局开关进入退役倒计时。

## 6. 风险与代价诚实清单

1. **状态派生误判**:纯问答会话没有「收工」概念,硬套五态会尴尬——速问逃生舱是必须品,不是锦上添花。
2. **叙事镜头质量天花板**:规则映射只能做到「动词+对象」级人话;模型生成摘要则费 token 且离线场景弱化。V3 前诚实按「工具动词人话化 + 时间线」交付,不承诺散文。
3. **专家直觉损失**:聊天优先用户被动迁移;经典布局必须覆盖 Preview 和至少 1 个公开过渡 Release,且新壳层专家模式的原始镜头必须一键直达；但默认切到新壳层后最迟第 2 个公开 Release 退出双壳,避免把迁移保护变成永久维护税。
4. **「需要你」后端成本是真的**:权限 prompt 状态化触碰回合生命周期,与 CLI 引擎 blocking 语义耦合;按本仓惯例应主树串行做,不派并行 worktree。
5. **双壳并存维护税**:并存期每个新功能两套入口;必须给并存设死线(建议 2 个 release)。
6. **文档全量重写**:USER-GUIDE 按三栏撰写,V1 上线即需重写第 4 章,教学成本前置。
7. **隐喻过度风险**:瓷章/班组若压过功能语义即装饰主义;规矩=隐喻只上不承载信息的装饰层,班组图节点数据全部来自既有字段。
8. **坞的信息密度下降**:72px 轨装不下会话搜索/分组;重度多会话用户(专家)依赖档案视图,多一跳——用 Ctrl+K 面板补偿(已有资产)。

## 7. 与两张设计图的对照

- `vnext-home.html`:案头条四事实、任务坞三态瓷章、交办箱(含速问逃生舱与本单安全档)、续办条(进行中 + 鎏金需要你)、熟活架、最近收活(带反悔柄)。
- `vnext-mission.html`:任务单头部(五态 + 工序段进度 + 用量)、镜头切换(叙事/班组图/原始,可点切换)、班组图(双引擎徽标、运行态呼吸、虚线鎏金拟增工序)、现场纪要、递话、收活台(需要你审批卡 + 成果 + 台账与整单回退)。

## 8. Pretender 3.0 北极星、目标与非目标

### 8.1 北极星

**从「我和 AI 聊了什么」转向「我交办了什么、现在进行到哪、需要我决定什么、最终交付了什么、能否反悔」。**

Pretender 的成功不以新页面数量衡量，而以用户是否能围绕一张任务单完成“交办—离场—回来监督—处理例外—验收—反悔—沉淀”的完整闭环衡量。

### 8.2 六个大版本目标

1. **Mission 是一级对象**：会话降为任务的沟通通道；一张任务单可以关联多次会话、多次 Agent Run 和多个执行引擎。
2. **一个全局「需要你」**：权限、计划、Agent 提问、任务池审批和冲突处理统一进入可恢复的 Intervention 收件箱。
3. **始终诚实的反悔能力**：区分可撤销、可恢复和不可逆；最近检查点、单步回退和整单回退能力始终在固定位置可见。
4. **镜头取代功能隐藏**：叙事、班组图、原始三个镜头共享同一任务事实，只改变表达密度，不制造“简易模式看不到功能”的墙。
5. **收工而不是聊天结束**：成果、验收、未完成项、变更、用量和审计在收工卡闭环，并可沉淀为熟活或记忆。
6. **复用 Escapade 的执行资产**：双引擎、工具、Agent Run、检查点、审计、技能、记忆和离线存储不因换壳重写。

### 8.3 明确非目标

- 不在 3.0 同时引入云端账号、多用户协作、移动端、遥测平台或新的大型工具宇宙。
- 不为“看起来像大版本”而重写双引擎、权限、工具调用、检查点和审计链路。
- 不把引入前端框架与 Pretender 绑定；继续保持零运行时依赖为默认约束，若要改变必须另立 ADR 与迁移收益证明。
- 不用模型生成的漂亮摘要代替权威状态；Mission 状态、待决策和回滚能力必须来自可验证的数据和事件。
- 不在 Pretender 立项前把概念名、版本号或兼容承诺混入 Escapade 的用户界面和发布物。

## 9. 数据地基：Mission Projection 与 Intervention

新壳层不能继续从当前会话、内存 pending Map 和页面可见卡片临时拼状态。Pretender 立项前必须建立两个稳定领域对象：

```text
Mission
├─ id / title / brief / state
├─ workspace / enginePolicy / permissionPolicy / budget
├─ sessionIds[] / runIds[]
├─ interventionIds[]
├─ outcomes[] / artifactRefs[] / changeSetRefs[]
├─ checkpointRefs[] / auditCursor
└─ createdAt / updatedAt / closedAt

Intervention
├─ id / missionId / runId? / nodeId?
├─ type: permission | question | plan | pool | conflict
├─ state: pending | resolved | expired | cancelled
├─ payload / decision / idempotencyKey
└─ createdAt / expiresAt? / resolvedAt?
```

### 9.1 实施约束

- **Projection，不是第二套状态机**：Mission 快照由 `session.mission`、Agent Run 事件、产物、变更、检查点、用量和审计引用投影而成；执行权仍在现有引擎与运行时。
- **未决事项必须持久化**：permission/question/plan/task-pool decision 不再只存在于进程内 Map；刷新和服务重启后必须能恢复、过期或安全拒绝。
- **决策必须幂等**：每个 Intervention 只有一次有效状态转换；重复请求返回既有结果，不重复批准工具或任务。
- **默认失败要安全**：恢复不了原 blocking continuation 时，状态转为 expired/cancelled 并保持未授权，不得因恢复机制放宽权限。
- **Quick Ask 是显式类型**：以用户动作或创建入口标识，不用文本关键词猜测；Quick Ask 可保留完整聊天体验而不强行制造任务收工语义。
- **回滚能力是事实字段**：任务只展示真实存在且仍有效的 checkpoint/change-set；外部副作用与不可逆动作单独列出。

### 9.2 兼容策略

- 旧 Session 首次读取时投影成 `Quick Ask` 或 `Legacy Mission`，不批量改写原文件；需要升级写入时使用显式 schema version。
- 新 Mission 保存对旧 session/run/checkpoint/audit 主键的引用，原 API 在双壳期继续工作。
- 经典壳层和新壳层消费同一聚合 API；禁止为两个壳层维护两套业务状态。
- 数据迁移、壳层切换和应用升级都必须有独立回退路径，不能把“切回经典布局”冒充数据回滚。

## 10. Pretender 分阶段实施

原 §5 的 V1/V2/V3 是 UI 置换顺序；正式工程实施按以下 P0—P4 门推进。P0 可用前端派生快速验证形态，但任何公开 Preview 前必须完成 P1 的持久化数据地基。

### P0 · 真实数据概念验证

- 在 Escapade 壳层中实现只读 Mission projection 原型和任务坞/案头条 PoC。
- 用单任务、长任务、多 Agent、Quick Ask 四条代表旅程验证五态、需要你、成果和反悔柄。
- 记录每个状态的权威来源和无法可靠派生的缺口；禁止用新增文案掩盖数据缺失。
- 产出交互验证、双壳成本、复用比例、性能基线和 P1 schema 草案。

### P1 · 领域基础

- 冻结 Mission / Intervention schema、事件语义、幂等规则、过期规则和迁移版本。
- 实现 `/api/missions`、单 Mission snapshot/events 和 Intervention list/decision API。
- 把现有 permission/question/plan/pool decision 接入持久化层，并补服务重启、超时、取消和重复决策测试。
- 建立旧 Session 适配器；不改变双引擎、工具、Agent Run、权限和检查点执行路径。

### P2 · 新壳层 Preview

- 上线案头条、任务坞、交办确认卡、Quick Ask 和全宽任务单。
- 第一阶段优先交付原始镜头，直接复用现有消息/工具卡渲染器；班组图和叙事镜头可逐步打开。
- 提供经典布局切换、迁移说明和故障回退；两个壳层读取同一 Mission API。
- 新功能只实现一个业务入口，通过共享 command 层由两个壳层调用，避免双份逻辑。

### P3 · 完整任务交付体验

- 上线全局「需要你」、班组图、叙事镜头、停工卡、成果/台账、单步与整单反悔柄。
- 收工卡整合验收、未完成项、产物、变更、成本和审计，并接入“存为熟活”“记住习惯”。
- 对长任务离场/回来、跨任务处理 Intervention、失败恢复和人工接管做专项可用性验证。

### P4 · 硬化与切换

- 完成性能、双主题、双语、无障碍、离线升级、数据迁移、恢复与安全红队。
- 新壳层成为默认后保留至少 1 个公开过渡 Release；最迟第 2 个公开 Release 退出双壳。
- 退出经典壳层前提供数据导出、版本回滚说明和最终兼容报告；原始镜头继续承担专家入口。

## 11. Pretender go / no-go 门

### 11.1 必须全部通过的 go 条件

1. **状态可信**：代表旅程的五态、成果和“需要你”均来自持久化事件/数据，不依赖 assistant 文本关键词猜测。
2. **干预可恢复**：所有未决事项刷新后仍存在；服务重启后能恢复等待或安全过期，不出现默认批准。
3. **决策幂等**：批准、拒绝、重试和取消的重复提交不会重复执行工具、节点或任务池变更。
4. **旧数据无损**：旧会话可打开、可继续、可切经典布局；迁移失败不会改坏原始会话。
5. **单一执行路径**：新壳层不引入第二套引擎、权限、工具执行、检查点或审计实现。
6. **反悔诚实**：界面能区分可完整回滚、部分恢复和不可逆副作用；展示的按钮都有可验证的实际能力。
7. **核心信息可达**：任务状态、需要你、成果和回滚入口最多两次操作到达。
8. **性能不过度退化**：基准数据集下首屏可交互目标 `<1.5s`、主要视图切换 P95 目标 `<200ms`，长任务事件流不引发主视图全量重建。
9. **双壳有退出路径**：兼容期、冻结范围、移除条件和责任人明确，最迟第 2 个公开过渡 Release 完成退出。

### 11.2 任一命中即 no-go / 退回 P0-P1

- Mission 状态仍主要依赖前端轮询多端点后临时拼装，或依赖模型文案推断。
- permission/question/plan 等任一高风险未决状态在服务重启后可能丢失或被误批准。
- 为新壳层复制了第二套业务状态、执行循环、权限判断或回滚逻辑。
- 老会话必须破坏性迁移才能打开，且没有可靠备份与恢复路径。
- 双壳没有退出时限，或每个新功能必须手写两套业务实现。
- PoC 只证明视觉新颖，未证明四条代表旅程在可理解性、干预恢复或收工闭环上优于 Escapade。

评审通过只代表可以冻结 Pretender 3.0 范围，不代表立即发布；范围冻结后仍需建立独立的安全、迁移、离线包和发布质量计划。
