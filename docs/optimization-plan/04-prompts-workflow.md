# 04 · 提示词优化与模型工作流程规范化方案

> 对应所有者确定方向 4。核心判断：**功能上提示词体系已相当完备（分层、围栏、防注入、按需装载俱全），欠的是工程化——外置、i18n、缓存稳定、预算闸、评测。改动提示词是行为漂移高风险操作，必须先建护栏，再动文本。**

### 1.2 模型工作流程现状

- **预算**：`TOOL_ITERATION_BUDGETS` 100/200/300/50（06:1114）；`isLongToolTask` 关键词启发式升档（06:1119-1128）；按进展事件动态扩展（06:1147-1151；09:1102-1113）。
- **循环保护**：主/子回合同签名连击 warn@3/abort@5（09:1023-1029/1322-1347；08:666）；DAG 节点 progressFingerprint + noProgressLimit（09:609-621）；Mission 停滞 K 轮降级（10:584-592）。
- **plan 模式**：首消息 `PLAN:` 暂停-批准-仅当回合放行（09:1168-1232/1418）。
- **检查**：质量门 schema + verdictPasses + vote/dedupe 确定性聚合 + JSON 修复管线（08:881-887/1045；09:491-503；06:992-1020）。
- **上下文治理**：窗口三级解析 + 超限学习只降不升（10:56-72）、EMA 自校准（10:107-124）、L1 蒸发 / L2 摘要重播种（10:490-543）。

## 2. 实施策略：先护栏，后文本，再机制

### Phase A · 评测与护栏建设（W46，先于一切文本改动）

1. **提示词快照测试**：对 `buildProviderSystemPrompt` 在固定假配置下的输出做分层快照（每层单独断言 + 总长断言）；Claude 侧对 `--append-system-prompt` 拼装与 stdin `<workbench-context>` 注入做同款。快照即"改动清单自动生成器"——任何提示词 diff 必须体现在快照更新里，review 可见。
2. **请求体落盘断言扩展**：`FAKE_CAPTURE_DIR` 范式（dev-harness 已有）推广到压缩摘要、质量门、playbook 起草、JSON 修复器四类辅助提示词。
3. **跨模型 A/B 评测夹具**：以 `model-tier-probe.js` 为底，建"提示词基准集"——10–20 个代表性任务脚本（工具使用规范、先读后改、Office 禁令、loop 自救、plan 触发），改提示词前后各跑一次对比通过率。i18n/README.md:99 已立原则"模型内容本地化须版本化、回归评测、安全审查"，提示词复用同款纪律。
4. **预算断言**：系统提示总长闸（见 Phase C）+ 每层预算的静态锁。

### Phase B · 提示词外置与 i18n（W47–W48）

1. **外置为资源文件**：散在 06/09/13 的提示词文本抽为 `app/prompts/` 目录下的结构化文件（按层一文件，中英双份），构建期随 manifest 拼接或运行期按 `config.locale` 加载；代码里只留拼装逻辑。沿用 locale 文档副本同步惯例与"双语键同交"lint 纪律。
2. **单一注册点**：`prompts/registry.js` 导出全部层 + 拼装顺序声明，`buildProviderSystemPrompt` 瘦身为纯装配器；版本号注入（`PROMPT_PACK_VERSION`），会话元数据记录所用提示词包版本——为 A-B 实验与问题回溯奠基。
3. **中英语义对齐审校**：不是逐字翻译，而是逐层双语等价（工具协议守则、桌面规程这类行为关键层优先）；英文母语级润色。
4. **坏味道顺手清**：G7 死代码与口径不一致修复。

### Phase C · 结构与缓存优化（W48）

1. **系统提示稳定化分层**：system 首条只放逐字节稳定层（身份/工具协议/桌面规程/风格）；易变层（账本 digest、能力层、当日日期等）移到 user 侧尾部消息或回合首条 user 消息的前缀块。保住上游 prefix cache（对长会话是实打实的成本与延迟优化）。注意验证各 provider 对多 system/消息序的兼容性，fake-openai 剧本补断言。
2. **Provider 侧总预算闸**：仿 cmd8191 思路做逻辑预算——总长超阈值时按层优先级降级（项目层截断 → 技能索引瘦身 → 账本 digest 化），并记录降级事件供 UI 展示（压缩🗜事件族已有先例）。
3. **工具 description 瘦身**：`orchestrate_agents` 等超长常驻描述压缩，细则挪到 tool_search 按需返回（06:1209-1211 的按需装载机制已有，把"描述细则"也挂进去）。
4. **技能/记忆注入策略复核**：技能 181 个 i18n key 是最大前缀，索引 3000 字预算在小窗模型下是否再分级，结合 EMA 校准数据（`context-calibration.json`）评估。

### Phase D · 模型工作流规范化（W48）

1. **语义级 loop-guard**：在"同签名连击"之外增加"结果指纹无进展"判定——工具结果摘要 hash 连续 N 次无新信息即 warn；把 DAG 节点的 progressFingerprint 思路推广到主回合（09:611 已在节点级验证）。误报防护：read/search 类探索工具放宽阈值。
2. **预算口径统一**：用户钳 [1,200] vs hardLimit 300（06:1132/1139）拉齐；`isLongToolTask` 关键词启发式升级为"首轮工具使用模式再判定"（首轮即大量 exec/read 自动升档），减少开局误判。
3. **回合级输出契约**：目前仅 DAG 节点有 outputSchema/质量门；给主回合引入轻量 self-check 约定（长任务收尾时输出"完成声明：做了什么/没做什么/验证方式"），先提示词约定后机械校验。
4. **Claude 引擎对齐**：角色 prompt 注入补齐（07:1722-1752 的 role 职责文本经 `--append-system-prompt` 或 stdin 注入子代理会话）；plan 模式差异写进能力对齐表（Claude 走 CLI 自带 plan mode）。
5. **《模型工作流规范》文档**：面向用户与贡献者写清"规划-执行-检查-循环保护-预算-压缩"的工作方式与各档预算语义，挂到 ruyi-workbench/docs/；规范即营销素材（"可审计的工作方式"）。

## 3. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 提示词改动引发模型行为漂移（工具误用/loop/拒答） | Phase A 护栏先行：快照 + A/B 基准集，任何文本改动过评测才合入 |
| 消息层移动（Phase C）破坏 provider 兼容性 | fake-openai 剧本 + 各主流 provider 抽测；失败回滚为原结构 |
| 双语提示词语义分叉 | 逐层等价审校 + 双语快照对照；行为关键层以中文原版为准绳翻译 |
| 语义 loop-guard 误伤正常探索 | 探索类工具宽阈值 + warn 先行不 abort + e2e 覆盖"换路径多读"场景 |
| 外置资源文件加载失败 | 构建期内联兜底或启动自检（syntax-gate 同款思路），缺文件拒绝启动并明示 |
